# گزارش بررسی فنی ریپو Agent.md-Generator

این گزارش با تمرکز بر «نقاط قوت/ضعف» و به‌صورت فایل‌به‌فایل برای کدهای اصلی تهیه شده است.

## 1) ریشه پروژه (Angular App)

### `package.json`
**نقاط قوت**
- اسکریپت `dev` با `cross-env` برای تزریق `GEMINI_API_KEY` و تنظیم host/port برای محیط توسعه مناسب است.
- وجود `lint` و `test` نشان می‌دهد چرخه کیفیت در نظر گرفته شده است.

**نقاط ضعف / ریسک**
- کلید API از طریق تعریف زمان build/runtime به سمت کلاینت می‌رود؛ این یعنی در عمل کلید روی مرورگر کاربر قابل مشاهده است و برای سرویس‌های پولی ریسک امنیتی/هزینه ایجاد می‌کند.
- وجود هم‌زمان Angular app در ریشه و Next.js app در `agentsmd/` بدون مستندات یکپارچه، نگه‌داری را مبهم می‌کند.

### `angular.json`
**نقاط قوت**
- بودجه باندل برای production تعریف شده و این برای کنترل وزن خروجی مفید است.
- پیکربندی lint و test مشخص است.

**نقاط ضعف / ریسک**
- مقدار پیش‌فرض `GEMINI_API_KEY` در `define` به شکل ثابت تعریف شده؛ هرچند placeholder است، اما معماری کلی همچنان «افشای کلید در سمت کلاینت» را تقویت می‌کند.

### `README.md`
**نقاط قوت**
- راه‌اندازی اولیه بسیار سریع و کوتاه است.

**نقاط ضعف**
- برای پروژه‌ای با هدف جامعه‌محور، README خیلی مینیمال است و درباره معماری، محدودیت‌های امنیتی، نمونه خروجی، تست، و ساختار دو اپ توضیح کافی نمی‌دهد.

## 2) کد Angular (`src/app/*`)

### `src/app/app.ts`
**نقاط قوت**
- استفاده از Angular Signals (`signal`) و `OnPush` باعث عملکرد بهتر UI و state management ساده‌تر شده است.
- مدیریت خطا با کلیدهای معنا‌دار (`API_KEY_MISSING`, `TIMEOUT`, ...) تجربه کاربری را بهتر می‌کند.
- خروجی چندفرمته (Markdown/HTML/PDF) ارزش محصول را بالا برده است.

**نقاط ضعف / ریسک**
- استفاده از `any` در چند بخش، تایپ‌سیفتی را کاهش می‌دهد.
- متد `onSubmit` چند مسئولیت هم‌زمان دارد (اعتبارسنجی فرم، آماده‌سازی داده، orchestration تماس AI، نگاشت خطا) و می‌تواند به سرویس/لایه‌های کوچک‌تر شکسته شود.
- تبدیل Markdown به HTML با `marked.parse` بدون sanitize برای فایل HTML دانلودی، ریسک تزریق محتوای ناامن را ایجاد می‌کند (خصوصاً اگر متن تولیدی model شامل HTML/JS ناخواسته باشد).

### `src/app/app.html`
**نقاط قوت**
- UI مدرن، یکپارچه و جذاب؛ استفاده از کلاس‌های utility برای سرعت توسعه خوب است.
- ساختار فرم و پیش‌نمایش برای کاربر واضح است.

**نقاط ضعف**
- فایل بسیار بزرگ و monolithic است؛ نگه‌داری، تست و بازاستفاده را سخت می‌کند.
- بهتر است به کامپوننت‌های کوچک‌تر تقسیم شود (Header/FormSection/PreviewSection/ActionButtons).

### `src/app/language.service.ts`
**نقاط قوت**
- دوزبانه بودن (fa/en) و پشتیبانی `dir` راست‌به‌چپ به‌خوبی پیاده شده است.
- نگه‌داری ترجمه‌ها به‌صورت ساختاریافته در یک محل انجام شده است.

**نقاط ضعف**
- فایل ترجمه با رشد پروژه بزرگ و پرهزینه می‌شود؛ بهتر است به resource file جدا (مثلاً `i18n/*.json`) منتقل شود.
- `UI_STYLES` و `TRANSLATIONS` در یک فایل، مسئولیت‌ها را مخلوط می‌کند.

### `src/app/theme.service.ts`
**نقاط قوت**
- پشتیبانی theme با تشخیص `prefers-color-scheme` و localStorage UX خوبی ایجاد کرده است.

**نقاط ضعف**
- منطق theme ساده و مناسب MVP است، اما abstraction برای تم‌های بیشتر یا design token system ندارد.

### `src/app/gemini.service.ts`
**نقاط قوت**
- ساختار prompt دقیق و با sectionهای مشخص است.
- مدیریت timeout و نگاشت خطاهای quota/network مناسب است.

**نقاط ضعف / ریسک**
- مدل `gemini-3.1-pro-preview` ممکن است ناپایدار/غیردائمی باشد؛ بهتر است fallback یا config-driven model id اضافه شود.
- prompt interpolation مستقیم از ورودی کاربر انجام می‌شود؛ برای robustness بهتر است delimitation/escaping مشخص‌تر انجام شود.
- نبود backend proxy باعث می‌شود امنیت و کنترل مصرف API محدود باشد.

### `src/app/app.spec.ts`
**نقاط قوت**
- وجود تست پایه برای ساخت کامپوننت.

**نقاط ضعف**
- پوشش تست بسیار پایین است؛ تست سرویس‌ها، رفتار فرم، خطاها، خروجی‌ها و دانلودها وجود ندارد.

## 3) زیرپروژه `agentsmd/` (Next.js)

### `agentsmd/package.json`
**نقاط قوت**
- ساختار استاندارد Next.js + TypeScript + Tailwind.

**نقاط ضعف**
- وجود دو اپ مستقل در یک ریپو (Angular + Next) بدون استراتژی monorepo روشن، باعث پیچیدگی onboarding می‌شود.

### `agentsmd/pages/index.tsx`
**نقاط قوت**
- تفکیک مناسب بخش‌های صفحه با کامپوننت‌های مجزا.
- کش حافظه‌ای برای کاهش rate-limit گیت‌هاب در dev ایده خوبی است.

**نقاط ضعف / ریسک**
- چندین fetch ترتیبی به GitHub انجام می‌شود؛ برای کارایی می‌تواند parallel با کنترل concurrency انجام شود.
- لاگ خطاها generic هستند و observability محدودی دارند.

## 4) جمع‌بندی معماری و محصول

### نقاط قوت کلان
- ایده محصول کاربردی و مشخص برای جامعه توسعه‌دهندگان.
- UX خوب، بخصوص برای کاربران فارسی‌زبان و خروجی چندفرمته.
- prompt engineering نسبتاً ساختاریافته و مفید.

### نقاط ضعف کلان (اولویت بالا)
1. **امنیت کلید API**: کلید در کلاینت استفاده می‌شود و باید به backend منتقل شود.
2. **پوشش تست پایین**: برای پروژه community-facing، ریسک regression بالاست.
3. **مرزبندی معماری نامشخص**: هم‌زیستی Angular و Next بدون روایت شفاف در مستندات.
4. **نگه‌داری UI**: فایل template اصلی بسیار بزرگ و monolithic.

## 5) پیشنهاد Roadmap بهینه‌سازی

1. انتقال فراخوانی Gemini به backend endpoint (server-side token handling + rate limit).
2. شکستن `app.html` و `app.ts` به چند feature component + facade service.
3. افزودن تست‌های هدفمند:
   - unit: `GeminiService`, `LanguageService`, `ThemeService`
   - integration: submit flow + error states
4. بهبود README:
   - معماری، نحوه اجرا برای هر اپ، امنیت، contribution guide.
5. بهبود کیفیت خروجی HTML/PDF با sanitize pipeline.
